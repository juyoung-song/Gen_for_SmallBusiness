`PRD.md`와 `architecture.md`가 **무엇을 만들지**를 정의한다면, 이 문서는 **어떤 사고 모델로 동작할지**를 정의한다.

## 1. 핵심 컨셉

이 시스템의 본질은 **"카페·베이커리·디저트 가게의 브랜드 일관성을 유지하면서 인스타 광고를 자동 생성"** 하는 것이다.

### 1.1 대상 사용자 (Scope)

본 MVP 의 사용자는 **카페·베이커리·디저트 가게 사장님** 으로 좁혀진다.
- 베이커리 (빵 전문)
- 카페 (커피/음료, 디저트 동반 OK)
- 디저트 가게 (마카롱, 케이크, 푸딩, 젤라토 등 디저트 특화)
- 위 카테고리를 결합한 매장 (예: 베이커리 카페, 디저트 카페)

이 외의 업종(뷰티, 패션, 의류, 자동차 등)은 본 MVP 범위 밖이다. 향후 확장 시
별도 카테고리 모듈을 추가하는 방식으로 풀이한다.

이렇게 좁힌 이유:
- 광고 목적 카테고리 6종(§4.1.1)이 본질적으로 카페/베이커리에 친화적
- 이미지 생성 모델의 출력 안정성이 좁은 도메인에서 훨씬 높음
- 카피 톤·색감·사진 스타일에 대한 도메인 가정이 명확해짐
- 사용자가 잘못된 레퍼런스 URL(예: K-뷰티)을 넣어도 시스템이 카테고리를 고정해
  잘못된 광고가 나오는 것을 방지할 수 있음

### 1.2 동시에 만족해야 할 두 축

소상공인이 인스타 마케팅을 시작할 때 가장 어려워하는 것은 **개별 광고를 만드는 것이 아니라, 매번 비슷한 톤앤매너를 유지하는 것**이다. 따라서 시스템은 다음 두 축을 동시에 만족해야 한다:

1. **즉시성**: 상품 정보를 넣으면 곧바로 인스타 게시물이 나와야 한다
2. **일관성**: 매번 만들어지는 광고가 같은 브랜드의 것처럼 보여야 한다

이를 위해 시스템은 **온보딩 1회 + 매 광고 생성 시 N회**의 두 단계로 나뉜다.

## 2. 데이터 모델

### 2.1 핵심 자산

| 자산 | 정의 | 변경성 | 생성 시점 |
|------|------|--------|-----------|
| **brand_image.txt** | 브랜드 정체성 (system prompt 역할) | **불변** | 온보딩 1회 |
| **상품 (raw 이미지)** | 사용자가 직접 찍은 화장 전 사진 | 추가만 | 신상품 등록 시 |
| **참조 이미지 (업로드 사진)** | 시스템이 생성해서 인스타에 게시한 화장 후 결과물 | 추가만 | 매 광고 생성 후 |

### 2.2 상품과 참조 이미지의 관계

```
상품 (Product)
├─ name: "블루베리 치즈케이크"
├─ description: "프랑스산 크림치즈 사용..."
├─ raw_image: cake_raw.jpg            ← "화장 전" (사용자 업로드)
└─ generated_uploads: [                ← "화장 후" (시스템 생성, 인스타 게시 완료)
     { id, image, caption, posted_at, ... },
     { id, image, caption, posted_at, ... },
     ...
   ]
```

핵심 관계:
- **raw 이미지 = "화장 전"** (재료)
- **참조 이미지 = "화장 후"** (결과물)
- 두 개는 같은 상품으로 묶이는 부모-자식 관계

### 2.3 brand_image.txt 의 위상

`brand_image.txt`는 단순한 메타데이터가 아니라 **모든 생성 호출에 항상 주입되는 system prompt**이다.

특성:
- **불변**: 온보딩 시 한 번 결정되고, 이후 광고 생성 결과로 갱신되지 않는다
- **GPT 정제**: 사용자가 작성한 자유 텍스트 + 레퍼런스 인스타 분석 결과를 GPT가 종합해서 작성
- **단일 파일**: MVP 단계에서는 한 사용자/한 브랜드 가정

> ⚠️ **위험 신호**: brand_image.txt가 잘못 작성되면 모든 광고가 잘못된 톤으로 나온다. 따라서 온보딩 단계의 품질 보장이 시스템 전체 품질을 좌우한다. → §3.1의 "검수 단계" 참고.

## 3. 온보딩 플로우 (1회)

### 3.1 흐름

```
[사용자 입력]
  1. 자유 텍스트 — "어떤 브랜드인가요?"
     · 감성, 타깃, 색감, 회피 사항 등 자유 서술
  2. 레퍼런스 인스타 프로필 URL (1개, 추후 N개로 확장 가능)
     · 추구미: 벤치마크하고 싶은 인스타 계정
       │
       ▼
[자동 처리]
  3. 인스타 프로필 헤드리스 캡처
     · scripts/insta_screenshot.py (browser-use CLI 기반)
     · 1~2장만 캡처 (한 화면에 게시물 다수 노출되므로 충분)
       │
       ▼
  4. GPT Vision 분석 + 종합
     · input: 사용자 자유 텍스트 + 캡처 이미지(들)
     · output: 정제된 system prompt 형식의 brand_image.txt
       (색감, 무드, 톤, 회피 사항 등 구조화)
       │
       ▼
[검수]
  5. 사용자 검수 (타협 모드)
     · 결과 표시 + "그대로 OK" 큰 버튼 (디폴트 강조)
     · 작은 "수정하기" 버튼 (까다로운 사용자용)
       │
       ▼
[저장]
  6. brand_image.txt 확정 → 로컬 DB에 저장 (이후 불변)
```

### 3.2 결정 사항

| 항목 | 결정 | 근거 |
|------|------|------|
| brand_image.txt 작성 방식 | GPT 자동 생성 + 사용자 검수 | 소상공인이 자유 텍스트로 톤을 표현하기 어려움 |
| 레퍼런스 URL 단위 | 프로필 URL (1개) | MVP 단순화. 게시물 단위는 v1.5 |
| 캡처 장수 | 1~2장 | 인스타 그리드 특성상 한 화면에 게시물 다수 노출 |
| 검수 단계 | 타협 모드 | 안전성 + 빠른 진입의 균형 |
| 다중 사용자 | 단일 사용자 가정 | MVP 범위 외, 멀티테넌트는 후순위 |

## 4. 광고 생성 플로우 (매번)

### 4.1 입력

| 입력 | 유형 | 필수 여부 | UI |
|------|------|-----------|-----|
| 상품명 | 텍스트 | 필수 | 드롭다운/갤러리 (기존 상품 선택) |
| 상품 설명 | 텍스트 | 필수 | 텍스트 입력 |
| 광고 목적 | 카테고리 + 자유 텍스트 | 필수 | 칩 버튼 6종 + 자유 텍스트 입력 |
| 신상품 토글 | 불리언 | 필수 | 토글 |
| raw 이미지 업로드 | 파일 | **신상품 토글 ON 시 필수** | 업로드 |
| 참조 이미지 선택 | 파일 ID 리스트 | 옵션 | 갤러리 (전체 풀에서 다중 선택) |

#### 4.1.1 광고 목적 카테고리 6종

칩 버튼 형태로 노출되며, 사용자는 1개 선택 후 자유 텍스트로 보강한다.

1. 신메뉴 출시
2. 주말/시즌 한정
3. 할인/이벤트
4. 일상/감성 (메뉴 아닌 분위기)
5. 영업 안내 (휴무, 시간 변경)
6. 감사/안부 (1주년, 명절 인사)

> 카테고리별 프롬프트 템플릿이 텍스트 생성 단계에서 활용된다.

#### 4.1.2 신상품 토글의 동작

- **ON**: raw 이미지 업로드란이 노출 + 필수화. 생성 후 백그라운드로 product 테이블에 INSERT
- **OFF**: 상품 드롭다운에서 선택한 상품의 raw 이미지를 DB에서 자동 로드
- **폴백**: 토글 OFF + 드롭다운 미선택 시 → 폼 검증으로 차단

#### 4.1.3 참조 이미지 선택 — 옵션이지만 강력함

- **풀 범위**: 전체 (특정 상품에 묶이지 않음)
  → 사용자는 "이 베이커리의 과거 광고 톤" 전체에서 영감을 고를 수 있음
- **신상품의 첫 광고**일지라도 **다른 상품의 과거 결과물**을 참조할 수 있음
  → 신상품 첫 광고부터 브랜드 톤이 이어짐
- **UI**: 썸네일 갤러리 (드롭다운 불가능 — 시각적 결정이므로)

### 4.2 컨텍스트 조립

광고 생성 직전에 다음을 모두 모은다:

```
context = {
    system_prompt: brand_image.txt,        # 항상
    product_name: input.상품명,             # 항상
    product_description: input.상품설명,    # 항상
    product_raw_image: 신상품 ? 업로드 : DB,  # 항상 1장
    reference_images: input.참조이미지[],   # 옵션 (0~N장)
    goal_category: input.광고목적.카테고리,  # 항상
    goal_freeform: input.광고목적.자유텍스트, # 항상
}
```

### 4.3 생성 단계 (병렬)

```
context
  │
  ├─→ [VM - nano banana]  (이미지 생성기)
  │     → output: 광고 이미지
  │
  └─→ [VM - GPT]          (텍스트 생성기)
        → output: 광고 카피 + 캡션 + 해시태그
```

두 호출은 **동시 실행**된다 (병렬). 결과가 모두 모이면 사용자에게 표시.

### 4.4 사용자 대기 시간 최소화 (Q4 결정 — 하이브리드)

```
T=0ms   [사용자 클릭]
T=10ms  [업로드 파일을 staging 디렉토리에 즉시 저장]   ← 동기, 손실 방지
T=50ms  [생성기 호출 시작]                            ← 사용자 대기 시작
...
T=Xms   [결과 표시]                                  ← 사용자 대기 끝
        │
        ▼ [백그라운드, 사용자는 결과를 보고 있음]
        - DB row 생성 (product / generated_upload)
        - staging → permanent 파일 이동
        - 신상품이면 product 테이블 INSERT
        - 인스타 업로드 성공 시 generated_upload 테이블 INSERT
```

핵심 원칙:
- **파일은 즉시** staging에 저장 (손실 방지)
- **DB row는 생성 후** 백그라운드에서 (사용자 대기 시간에 영향 X)
- **인스타 업로드 후**에만 generated_upload로 승격 (참조 이미지 풀 = 게시 완료된 것만)

### 4.5 인스타 자동 게시

- 자동 게시 (사전 준비 완료)
- Meta Graph API + 사업자 인증 + 토큰 사전 발급된 상태
- 게시 성공 시: generated_upload 테이블에 INSERT → 다음 생성의 참조 이미지 풀로 편입
- 게시 실패 시: 결과는 보여주되 백그라운드 재시도

## 5. 데이터베이스 스키마 (구상)

> 정확한 스키마는 구현 단계에서 확정. 여기서는 개념 모델만.

```
brand_image
  ├─ id (PK)
  ├─ user_id (FK, MVP에선 단일값)
  ├─ content (TEXT, system prompt)
  ├─ source_freetext (TEXT, 사용자 원본 입력)
  ├─ source_reference_url (TEXT, 레퍼런스 인스타 URL)
  ├─ source_screenshots (JSON, 캡처 파일 경로 리스트)
  └─ created_at

product
  ├─ id (PK)
  ├─ name (UNIQUE per user)
  ├─ description
  ├─ raw_image_path
  └─ created_at

generated_upload
  ├─ id (PK)
  ├─ product_id (FK → product.id)
  ├─ image_path
  ├─ caption
  ├─ goal_category
  ├─ goal_freeform
  ├─ instagram_post_id (게시 후 발급)
  ├─ posted_at
  └─ created_at
```

`brand_image`는 사실상 1행만 존재하지만, 향후 다중 사용자 확장을 위해 테이블로 둔다.

## 8. 리팩터링 베이스로서의 위치

이 문서는 [refactor/won/main](#) 브랜치의 시작점이다. 이 브랜치는 **2c808e0** (2026-04-06 dev 머지)에서 분기되었으며, 그 시점 이후 [enhance/won/img-generation](#) 등에서 진행된 기능들(배경 교체, FLUX fallback, 인스타 스크린샷 등)은 본 설계에 맞게 **재통합**될 예정이다.

기능별 작업 브랜치는 다음과 같이 활용한다:

| 기존 브랜치 | 담은 작업 | 통합 방향 |
|-------------|-----------|-----------|
| `feature/won/insta-snapshot` | browser-use CLI 기반 인스타 캡처 | 온보딩 단계의 캡처 모듈로 통합 |
| `enhance/won/img-generation` | 배경 교체, FLUX fallback | 광고 이미지 생성 옵션으로 통합 |
| 기타 기능 브랜치 | (각자) | 본 설계의 컴포넌트에 매핑 |

리팩터링의 목표는 **새 기능 추가가 아니라 위 결정 사항들에 맞춰 기존 코드를 정리하는 것**이다.
